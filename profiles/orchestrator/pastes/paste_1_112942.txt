# Node.js ETH/BSC/TRON 多链自动归集系统最终技术方案 V1

## 版本定位

本方案定位为：

```text
单机生产稳定版 V1
```

核心目标：

```text
不引入 Redis
不引入 MQ
不引入 KMS/HSM
不做区块事件监听
不做签名服务拆分
先用 Node.js + MySQL + RPC 方式实现稳定、可追踪、可恢复的三链归集系统
```

支持范围：

```text
ETH 主网
BSC 链
TRON/TRX 链

ETH 主币归集
BNB 主币归集
TRX 主币归集

ERC20-USDT 归集
BEP20-USDT 归集
TRC20-USDT 归集
```

# 1. 总体架构

## 1.1 数据库分离

系统分为两个数据库：

```text
业务库 cadd
  └── 只保存业务地址和地址使用状态
      不保存私钥
      不保存归集状态

归集库 collector
  └── 保存加密私钥
      保存归集任务
      保存交易流水
      保存余额缓存
      保存 nonce 状态
      保存 RPC 节点
      保存 Gas 策略
      保存系统告警和状态日志
```

---

## 1.2 业务库职责

业务库已有表：

```sql
tl_trade_address
```

核心字段：

```text
id              业务地址 ID
address         用户充值地址
user_id         用户 ID
coin            币种，通常 USDT
chain           链，ETH/BSC/TRX/TRON
status          地址状态
```

状态含义：

```text
status = 0：地址未使用，不监听
status = 1：地址已分配/已使用，需要归集程序监听
```

业务库不保存：

```text
私钥
归集任务状态
tx_hash
补手续费状态
链上确认状态
```

---

## 1.3 归集库职责

归集库负责：

```text
1. 生成钱包地址和私钥
2. 使用 AES-256-GCM 加密私钥
3. 将地址提供给业务库使用
4. 同步业务库 status=1 的地址
5. 轮询扫描地址余额
6. 创建归集任务
7. 自动补 ETH/BNB/TRX 手续费
8. 执行 ETH/BSC/TRON 主币和 USDT 归集
9. 管理 EVM nonce
10. 管理 RPC 节点池
11. 管理 Gas 策略
12. 监控交易确认
13. 处理 pending 超时
14. 记录任务状态日志
15. 记录系统告警
```

# 3. 第一版核心设计原则

## 3.1 保持架构克制

第一版不引入：

```text
Redis
BullMQ
Kafka
RabbitMQ
KMS
HSM
MPC
签名服务
区块事件监听
TRON 能量租赁
多机分布式锁
```

原因：

```text
先保证系统稳定上线
降低运维复杂度
降低开发风险
先解决最核心的归集闭环
```

---

## 3.2 第一版必须具备的生产安全垫

虽然不引入复杂中间件，但以下能力必须有：

```text
1. AES-256-GCM 私钥加密
2. MySQL 任务锁
3. collect_task_open 幂等控制
4. EVM Nonce Manager
5. RPC Pool 简化版
6. RPC 限流
7. Gas 策略
8. Pending 超时处理
9. 链上交易完整记录
10. 任务状态变更日志
11. 系统告警
```

# 6. 核心流程

## 6.1 地址生成流程

```text
归集系统生成 ETH/BSC/TRON 地址和私钥
    ↓
私钥使用 AES-256-GCM 加密
    ↓
密文私钥保存到 collector.wallet_key
    ↓
地址写入业务库 cadd.tl_trade_address，status=0
    ↓
业务系统分配地址给用户
    ↓
业务库将 status 更新为 1
```

---

## 6.2 地址同步流程

```text
addressSyncJob 定时查询业务库 tl_trade_address
    ↓
筛选 status = 1 的地址
    ↓
标准化 chain 和 address
    ↓
在 collector.wallet_key 中查找 chain + address
    ↓
找到私钥记录：
      写入/更新 wallet_address，status=1
    ↓
找不到私钥记录：
      写入/更新 wallet_address，status=3 缺私钥
      写入 system_alert
```

---

## 6.3 余额扫描流程

```text
balanceScanJob 查询 wallet_address.status=1 的地址
    ↓
按 chain 分批扫描
    ↓
通过 rpcService 获取可用 RPC
    ↓
通过限流器限制请求频率
    ↓
查询主币余额
    ↓
查询 USDT 余额
    ↓
更新 address_balance
    ↓
达到归集阈值则创建 collect_task
    ↓
通过 collect_task_open 防止重复任务
```

---

## 6.4 USDT 归集流程

```text
发现 USDT 余额达到阈值
    ↓
创建 TOKEN_COLLECT 任务
    ↓
collectJob 领取任务并加锁
    ↓
检查该地址主币余额是否足够支付手续费
    ↓
主币不足：
      任务状态改为 1 待补手续费
    ↓
feeJob 给用户地址补 ETH/BNB/TRX
    ↓
补手续费交易广播
    ↓
monitorJob 确认补手续费成功
    ↓
任务状态改为 3 待归集
    ↓
collectJob 解密用户地址私钥
    ↓
发起 USDT 转账到归集地址
    ↓
monitorJob 确认归集成功
    ↓
任务完成，释放 collect_task_open
```

---

## 6.5 主币归集流程

```text
发现 ETH/BNB/TRX 余额达到阈值
    ↓
创建 MAIN_COLLECT 任务
    ↓
计算可归集金额
    ↓
ETH/BNB：
      可归集金额 = 当前余额 - 预计 gas - reserve_main_amount
    ↓
TRX：
      可归集金额 = 当前余额 - reserve_main_amount
    ↓
解密用户地址私钥
    ↓
发起主币转账到归集地址
    ↓
写入 chain_transaction
    ↓
monitorJob 确认成功
    ↓
任务完成
```

# 7.1 地址私钥表：`wallet_key`

```sql
CREATE TABLE `wallet_key` (
  `id` BIGINT NOT NULL AUTO_INCREMENT,
  `chain` VARCHAR(32) NOT NULL COMMENT 'ETH/BSC/TRON',
  `address` VARCHAR(128) NOT NULL,
  `address_hex` VARCHAR(128) DEFAULT NULL COMMENT 'TRON hex地址',

  `private_key_cipher` TEXT NOT NULL COMMENT 'AES-256-GCM密文',
  `private_key_iv` VARCHAR(64) NOT NULL COMMENT '12字节IV hex',
  `private_key_tag` VARCHAR(64) NOT NULL COMMENT 'GCM认证tag hex',
  `encryption_algo` VARCHAR(64) NOT NULL DEFAULT 'AES-256-GCM',
  `key_version` INT NOT NULL DEFAULT 1,

  `status` TINYINT NOT NULL DEFAULT 1 COMMENT '1正常 0禁用 2冻结',
  `created_at` INT NOT NULL,
  `updated_at` INT NOT NULL,

  PRIMARY KEY (`id`),
  UNIQUE KEY `uniq_chain_address` (`chain`, `address`),
  KEY `idx_status` (`status`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

# 9. 私钥加密方案

## 9.1 加密算法

```text

```

---

## 9.2 密钥来源

```text

```

要求：

```text
32 字节
64 位 hex 字符串
不得入库
不得提交 Git
不得打印日志
```

生成方式：

```bash

```

---

## 9.3 加密字段

```text
private_key_cipher  密文
private_key_iv      12 字节随机 IV
private_key_tag     GCM 认证标签
```

---

## 9.4 加解密代码

```js
const crypto = require('crypto');

const ALGO = 'aes-256-gcm';

function getMasterKey() {
  const keyHex = process.env.WALLET_MASTER_KEY;

  if (!keyHex) {
    throw new Error('WALLET_MASTER_KEY missing');
  }

  const key = Buffer.from(keyHex, 'hex');

  if (key.length !== 32) {
    throw new Error('WALLET_MASTER_KEY must be 32 bytes hex');
  }

  return key;
}

function encryptPrivateKey(privateKey) {
  const key = getMasterKey();
  const iv = crypto.randomBytes(12);

  const cipher = crypto.createCipheriv(ALGO, key, iv);

  const encrypted = Buffer.concat([
    cipher.update(privateKey, 'utf8'),
    cipher.final()
  ]);

  const tag = cipher.getAuthTag();

  return {
    private_key_cipher: encrypted.toString('hex'),
    private_key_iv: iv.toString('hex'),
    private_key_tag: tag.toString('hex'),
    encryption_algo: 'AES-256-GCM',
    key_version: 1
  };
}

function decryptPrivateKey(row) {
  const key = getMasterKey();

  const decipher = crypto.createDecipheriv(
    ALGO,
    key,
    Buffer.from(row.private_key_iv, 'hex')
  );

  decipher.setAuthTag(Buffer.from(row.private_key_tag, 'hex'));

  const decrypted = Buffer.concat([
    decipher.update(Buffer.from(row.private_key_cipher, 'hex')),
    decipher.final()
  ]);

  return decrypted.toString('utf8');
}

module.exports = {
  encryptPrivateKey,
  decryptPrivateKey
};
```

# 14. EVM Nonce Manager 设计

## 14.1 使用范围

需要 nonce 管理的链：

```text

BSC
```

需要 nonce 管理的地址：

```text
1. 用户地址发 ETH/BNB 主币归集
2. 用户地址发 ERC20/BEP20-USDT 归集
3. 手续费钱包发 ETH/BNB 补手续费
```

---

## 14.2 分配规则

发送交易前：

```text
1. 开启 MySQL 事务
2. SELECT nonce_state FOR UPDATE
3. 获取链上 pending nonce
4. nonce = max(local_next_nonce, chain_pending_nonce)
5. 更新 next_nonce = nonce + 1
6. 提交事务
7. 使用该 nonce 构造交易
```

---

## 14.3 伪代码

```js
async function allocateNonce(chain, address, provider, conn) {
  const now = Math.floor(Date.now() / 1000);

  let row = await conn.queryOne(`
    SELECT *
    FROM nonce_state
    WHERE chain = ? AND address = ?
    FOR UPDATE
  `, [chain, address]);

  const chainPendingNonce = await provider.getTransactionCount(address, 'pending');
  const chainLatestNonce = await provider.getTransactionCount(address, 'latest');

  if (!row) {
    await conn.execute(`
      INSERT INTO nonce_state
      (
        chain, address, next_nonce,
        last_chain_nonce, last_pending_nonce,
        status, created_at, updated_at
      )
      VALUES (?, ?, ?, ?, ?, 1, ?, ?)
    `, [
      chain,
      address,
      chainPendingNonce + 1,
      chainLatestNonce,
      chainPendingNonce,
      now,
      now
    ]);

    return chainPendingNonce;
  }

  const localNextNonce = row.next_nonce ?? chainPendingNonce;
  const nonce = Math.max(Number(localNextNonce), Number(chainPendingNonce));

  await conn.execute(`
    UPDATE nonce_state
    SET next_nonce = ?,
        last_chain_nonce = ?,
        last_pending_nonce = ?,
        updated_at = ?
    WHERE id = ?
  `, [
    nonce + 1,
    chainLatestNonce,
    chainPendingNonce,
    now,
    row.id
  ]);

  return nonce;
}
```

## 15.2 `balanceScanJob`

职责：

```text
扫描启用地址的主币和 USDT 余额
```

扫描策略：

```text
1. 按 chain 分批
2. 按 last_scan_time 升序
3. 每批 ETH 100 个、BSC 200 个、TRON 100 个
4. 每条链独立限流
5. 扫描完成更新 last_scan_time
6. 更新 address_balance
7. 达到阈值则创建 collect_task
```

---

## 15.3 `collectJob`

职责：

```text
执行主币归集和 Token 归集
```

处理状态：

```text
status = 0 待处理
status = 3 待归集
```

流程：

```text
1. 领取任务并加 MySQL 锁
2. 重新查询链上余额
3. 判断是否仍满足归集条件
4. 如果是 Token，检查主币手续费余额
5. 手续费不足，任务改为 1 待补手续费
6. 手续费充足，解密用户私钥
7. EVM 链分配 nonce
8. 获取 Gas 参数
9. 发送交易
10. 写 chain_transaction
11. 更新 collect_task 状态
12. 写 task_status_log
```

## 15.4 `feeJob`

职责：

```text
给 Token 归集地址补手续费
```

处理状态：

```text
status = 1 待补手续费
```

流程：

```text
1. 领取待补手续费任务
2. 查询对应链 fee_wallet
3. 查询 fee_wallet 余额
4. 如果手续费钱包余额不足，写 system_alert
5. 解密 fee_wallet 私钥
6. EVM 链分配 fee_wallet nonce
7. 发送 ETH/BNB/TRX 到用户地址
8. 写 chain_transaction
9. 更新任务为 2 补手续费中
10. 写 task_status_log
```

---

## 15.5 `monitorJob`

职责：

```text
监控已广播交易确认状态
```

处理对象：

```text
chain_transaction.status = 1
```

流程：

```text
1. 查询已广播交易
2. 通过 rpcService 查询交易 receipt / transaction info
3. 判断交易是否成功
4. 判断确认数是否达到 confirm_blocks
5. 补手续费交易成功：
      collect_task.status = 3
6. 归集交易成功：
      collect_task.status = 6
      删除 collect_task_open
7. 交易失败：
      chain_transaction.status = 3
      collect_task 进入失败或重试
8. 当前时间超过 expire_at：
      chain_transaction.status = 4
      写 system_alert
```

## 15.6 `retryJob`

职责：

```text
处理可重试任务
```

可重试：

```text

网络异常
临时查询失败
手续费不足后已补充
```

不可直接重试：

```text

地址缺私钥
链上交易明确失败
```

```text

第 2 次：300 秒
第 3 次：900 秒
第 4 次：3600 秒

```

# 29. 最终结论

本 V1 方案是一个：

```text
Node.js + MySQL + ethers + tronweb
单机生产稳定版多链归集系统
```

它不追求一开始就做成复杂的钱包中台，而是优先实现：

```text
安全存储私钥
稳定扫描余额
可靠创建任务
自动补手续费
自动归集
交易确认监控
EVM nonce 管理
RPC 容灾
RPC 限流
Gas 策略
Pending 超时告警
任务幂等和审计
```

相比普通 Demo 归集程序，本方案已经补齐了第一版生产环境最容易出事故的关键点：

```text
nonce 冲突
RPC 单点故障
RPC 过载
gas 不可控
pending 交易卡死
任务重复执行
交易状态不可追踪
私钥明文存储
```

因此，最终第一版推荐定位为：

```text
单机可生产
MySQL 驱动
任务轮询型
三链自动归集系统
```

后续再根据业务规模逐步升级到：

```text
队列化
事件驱动
签名服务化
KMS/HSM
多机高可用
```
