下面是最终版**可执行、可落地、可开发的完整技术方案**。这版按你最新的方向重新整理：参考 X 帖子的“少量高质量交易 + AI 复盘 + Hermes 调度 + Telegram 通知”思路，但不照搬夸张收益叙事；界面采用你附件那种**黑绿赛博技术驾驶舱风格**，不是传统 Metabase 风格。

# 一、项目定位

项目名称建议：

```text
polymarket-hermes-agent-cockpit
```

项目目标：

```text
1. 自动采集 Polymarket 市场数据
2. 用 Rust 监听实时订单簿和价格变化
3. 用 Python/FastAPI 做 AI 策略分析、风控、模拟盘和报告
4. 用 Hermes 做多 Agent 调度、日报、复盘、Telegram 通知
5. 用 Next.js 做黑绿赛博风 Dashboard
6. 用 PostgreSQL 保存审计数据
7. 用 Redis Streams 做内部事件队列
8. 第一阶段只跑 Paper Trade，不自动真钱下注
```

Polymarket 官方文档显示，Gamma API 和 Data API 是公开接口，不需要认证；CLOB API 有公开的订单簿、价格接口，也有需要认证的订单管理接口，所以第一版只做只读数据和模拟盘是最稳的。([docs.polymarket.com][1])

# 三、核心设计原则

## 1. 不做“自动暴富机器人”

X 帖子的参考价值在于：

```text
少量交易
读订单簿
算概率差
AI 复盘
Hermes 调度
Telegram 通知
```

但不能照搬：

```text
500 美元变 88 万美元
30 分钟无需编码
AI 自动改 .env
全自动真钱下注
忽略合规和地区限制
```

## 2. 第一阶段只做 Paper Trade

```text
TRADE_ENABLED=false
PAPER_TRADE_ONLY=true
ENABLE_ORDER_EXECUTION=false
```

即使后期加入交易执行，也必须先走：

```text
AI 生成信号 → 风控通过 → 用户人工确认 → 系统执行限价单
```

Polymarket CLOB 是链下撮合、链上结算的混合系统，订单是 EIP-712 签名消息；涉及交易时必须处理认证、签名、订单生命周期和风控，所以第一版不应接私钥。([docs.polymarket.com][3])

## 1. `dashboard-web`：Next.js 前端

Next.js App Router 支持在 Server Components 中通过 `fetch`、ORM 或数据库直接获取数据，也适合把 API key、token 等敏感信息放在服务端而不是浏览器端。([nextjs.org][4])

## 2. `strategy-api`：Python 策略服务

职责：

```text
1. 市场基础信息采集
2. CLOB REST 价格采集
3. AI 分析市场规则、新闻和概率
4. 计算 edge / effective_edge
5. 执行风控规则
6. 生成 SKIP / WATCH / PAPER_TRADE 信号
7. 创建模拟交易
8. 生成日报、周报、复盘
9. 提供 Dashboard API
10. 消费 Redis Streams 实时行情事件
```

## 3. `market-engine-rs`：Rust 实时行情引擎

职责：

```text
1. 连接 Polymarket WebSocket
2. 订阅重点 market asset IDs
3. 接收 orderbook snapshot
4. 接收 price_change / last_trade_price
5. 在内存维护订单簿
6. 计算 midpoint / spread / depth / imbalance
7. 写入 Redis Streams
8. 写入 Redis 最新盘口 Hash
9. 记录延迟 metrics
```

Rust 只做速度敏感部分，不做 AI 分析，不做交易决策。

## 4. `hermes`：AI 调度与复盘

Hermes 负责：

```text
1. 每 30 分钟触发市场扫描
2. 每天生成早报
3. 每天生成晚间复盘
4. 调用 Strategy API
5. 通过 Telegram 推送信号和复盘
6. 管理 Market Scanner、Risk Manager、Journal Agent 等多 Agent
```

Hermes 官方描述其具备内置学习循环、技能沉淀和跨会话记忆，并可运行在 VPS、GPU 集群或 serverless 环境，也支持通过 Telegram 与云端 Agent 交互，适合做这个系统的调度层。([GitHub][5])

# 六、Redis Streams 设计

你不需要 GCP Pub/Sub。这里用的是 Redis 自带的 Streams。

Redis Streams 支持以流的形式追加记录，`XREADGROUP` 支持 consumer group 消费，适合多个 worker 可靠消费行情事件。([Redis][6])

## 1. Stream 名称

```text
realtime_market_events
```

## 2. Rust 写入事件

```text
XADD realtime_market_events *
  market_id 123
  token_id xxx
  best_bid 0.31
  best_ask 0.34
  midpoint 0.325
  spread 0.03
  bid_depth_1pct 12000
  ask_depth_1pct 9800
  imbalance 0.62
  event_time 2026-05-26T12:00:00Z
```

## 3. Python 消费事件

```text
XREADGROUP GROUP strategy_group strategy_worker_1
  COUNT 50
  BLOCK 5000
  STREAMS realtime_market_events >
```

## 4. Redis Key 规划

```text
realtime_market_events              # Redis Stream
market_snapshot:{token_id}          # 最新盘口 Hash
signal_lock:{market_id}             # 防重复生成信号
collector_lock                      # 防止多个采集任务重复跑
latency:{region}:{endpoint}         # 延迟监控缓存
```

```sql

    id BIGSERIAL PRIMARY KEY,
```

```sql
    side TEXT CHECK (side IN ('YES', 'NO')),

    created_at TIMESTAMPTZ DEFAULT now()
```


## 7. AI 参数建议表

```sql
CREATE TABLE strategy_config_suggestions (
    id BIGSERIAL PRIMARY KEY,
    key TEXT NOT NULL,
    old_value TEXT,
    suggested_value TEXT NOT NULL,
    reason TEXT,
    status TEXT CHECK (status IN ('PENDING', 'APPROVED', 'REJECTED', 'ROLLED_BACK')) DEFAULT 'PENDING',

```

这张表非常关键，用来替代 X 帖子里“AI 每晚自动改 `.env`”的危险做法。

# 八、策略与风控规则

## 1. 第一版策略

```text
短期事件赔率偏差策略
```

只做：

```text
1. 规则清晰
2. 结算时间 24 小时 - 7 天
3. 流动性足够
4. 点差足够小
5. 模型概率明显高于市场隐含概率
6. 能解释市场为什么可能定价错误
```

## 2. Edge 计算

```python
edge = ai_probability - market_price
effective_edge = ai_probability - market_price - spread - safety_margin
```

默认参数：

```text
SAFETY_MARGIN=0.03
MIN_EDGE=0.08
MIN_CONFIDENCE=0.65
MIN_LIQUIDITY_USD=10000
MAX_SPREAD=0.08
MAX_DAILY_SIGNALS=10
MAX_DAILY_PAPER_TRADES=5
```

## 3. 强制风控

```text
1. TRADE_ENABLED=false
2. PAPER_TRADE_ONLY=true
3. ENABLE_ORDER_EXECUTION=false
4. 每天最多 5 个 Paper Trade
5. 单个模拟仓位不超过模拟资金 1.25%
6. 连续亏损 3 单自动暂停
7. 日亏损超过 3% 自动暂停
8. 规则不清晰不做
9. 点差过大不做
10. 流动性不足不做
11. 涉及战争、恐袭、暗杀、军事、内幕信息直接跳过
12. AI 不能直接修改生产配置
13. AI 不能直接控制私钥
14. AI 不能自动真钱下注
```

# 十五、验收标准

## 第 1 周验收

| 指标                  |       目标 |
| ------------------- | -------: |
| 每日采集市场              |    > 500 |
| Rust WebSocket 稳定运行 |  > 12 小时 |
| Redis Stream 消息消费   |       正常 |
| AI 信号               | 3-10 个/天 |
| Paper Trade         |  1-5 个/天 |
| Dashboard 页面        |    全部可访问 |
| 登录/退出               |       正常 |
| 左侧菜单展开/收起           |       正常 |
| 自动真钱交易              |        0 |

## 第 30 天验收

| 指标     |        目标 |
| ------ | --------: |
| 模拟交易数量 |    >= 100 |
| 最大回撤   |     < 20% |
| 规则误读率  |      < 5% |
| 复盘完整率  |      100% |
| 模拟 ROI | 正数才考虑下一阶段 |
| 自动真钱交易 |      仍然禁用 |

Polymarket API 的限速由 Cloudflare throttling 执行，超过限制时请求会被延迟或排队，所以真正要优化延迟，不能只换语言，还要监控 REST/WebSocket 延迟、减少轮询、优先 WebSocket、避免触发限速。([docs.polymarket.com][7])

# 最终结论

这个项目最终定位应该是：

```text
Hermes Polymarket Agent Cockpit
一个预测市场 AI 投研、模拟盘、风控、复盘和参数审批驾驶舱
```

最终技术组合：

```text
Frontend：Next.js / TypeScript / Tailwind
AI Strategy API：Python / FastAPI
Realtime Engine：Rust
Queue / Cache：Redis Streams
Database：PostgreSQL
Agent：Hermes
Notification：Telegram
Deployment：Docker Compose
Mode：Paper Trade First
```

最重要的原则：

```text
先模拟，后真钱。
先复盘，后扩仓。
先验证策略，后优化延迟。
AI 可以建议，但不能自动改生产配置。
AI 可以筛选，但不能直接控制私钥和真钱。
```

这就是最终可执行、可落地的技术方案。
